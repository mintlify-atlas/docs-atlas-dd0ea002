Medplum gives you fine-grained control over who has access to your project, what data they can see, and how they are represented in FHIR. This guide covers the key concepts and common operations.

## Core concepts

### The user model

Three resources work together to represent a user's identity and access in Medplum:

| Resource | What it represents |
| --- | --- |
| `User` | A digital identity (email address). Exists above the project level. |
| `Project` | An isolated container of FHIR resources. |
| `ProjectMembership` | Grants a `User` access to a specific `Project`, with a profile and optional access policy. |

A single `User` can be a member of multiple projects with different roles in each.

### Server-scoped vs. project-scoped users

- **Server-scoped users** can be members of multiple projects. Typically used for administrators and developers.
- **Project-scoped users** exist within a single project. Typically used for clinicians and patients.

### Profiles

Within a project, every member is represented by a FHIR resource called their **profile**. The `ProjectMembership.profile` field links the membership to one of:

| Profile resource | Typical use |
| --- | --- |
| `Patient` | Patient portal access — grants access to the patient's own records |
| `Practitioner` | Clinical staff — typically has access to many patients' data |
| `RelatedPerson` | Family member or caregiver — access to a small number of patient records |

There are also programmatic profiles for non-human members:

| Profile resource | Typical use |
| --- | --- |
| `ClientApplication` | API keys for programmatic access |
| `Bot` | Event-driven custom functions |

## Inviting users

Only project admins can invite users. The invitation process creates the `User` (if they don't already exist), the appropriate FHIR profile resource, and a `ProjectMembership` linking them together.

### Via the Medplum App

Navigate to [app.medplum.com/admin/invite](https://app.medplum.com/admin/invite), fill in the user's details, select a profile type and optional access policy, and send the invitation.

### Via API

Use the [`/admin/projects/:projectId/invite`](https://www.medplum.com/docs/api/project-admin/invite) endpoint (project admins only):

<Tabs>
  <Tab title="TypeScript">
    ```typescript
    await medplum.post(`admin/projects/${projectId}/invite`, {
      resourceType: 'Practitioner',
      firstName: 'Jane',
      lastName: 'Doe',
      email: 'jane.doe@example.com',
    });
    ```
  </Tab>
  <Tab title="CLI">
    ```bash
    medplum post admin/projects/:projectId/invite \
    '{
      "resourceType": "Practitioner",
      "firstName": "Jane",
      "lastName": "Doe",
      "email": "jane.doe@example.com"
    }'
    ```
  </Tab>
  <Tab title="cURL">
    ```bash
    curl https://api.medplum.com/admin/projects/:projectId/invite \
      -H "Authorization: Bearer $TOKEN" \
      -H "Content-Type: application/json" \
      -d '{
        "resourceType": "Practitioner",
        "firstName": "Jane",
        "lastName": "Doe",
        "email": "jane.doe@example.com"
      }'
    ```
  </Tab>
</Tabs>

<Warning>
Do not delete `Patient`, `Practitioner`, or `RelatedPerson` resources that are linked to a `ProjectMembership`. Doing so will break the associated login. If you accidentally delete one, you will need to register a new project.
</Warning>

## Access policies

Access policies control what FHIR resources a project member can read or write. You can assign an access policy at invite time:

```typescript
await medplum.post(`admin/projects/${projectId}/invite`, {
  resourceType: 'Practitioner',
  firstName: 'Jane',
  lastName: 'Doe',
  email: 'jane.doe@example.com',
  membership: {
    access: [
      {
        policy: { reference: 'AccessPolicy/YOUR_POLICY_ID' },
      },
    ],
  },
});
```

See [Access Policies](https://www.medplum.com/docs/access/access-policies) for full documentation on writing and managing policies.

## Patient-level access for patient portals

When a user's profile is a `Patient` resource, access policies can be written to restrict that user to only their own data. This is the standard pattern for patient portal applications.

Link the user's `ProjectMembership` to the corresponding `Patient` resource, then write an access policy that uses the `%patient` variable to scope all FHIR queries to that patient's records.

## Open patient registration

You can allow patients to self-register for your project without an admin invitation. This is useful for patient portals where patients sign themselves up.

See the [Open Patient Registration](https://www.medplum.com/docs/user-management/open-patient-registration) guide for configuration steps.

## External IDs

External IDs let you link a Medplum user to an identifier in an external system (an EHR, a CRM, a legacy database, etc.). Set the `externalId` field on the invite payload:

```typescript
await medplum.post(`admin/projects/${projectId}/invite`, {
  resourceType: 'Patient',
  firstName: 'Alice',
  lastName: 'Smith',
  email: 'alice@example.com',
  externalId: 'EHR-PATIENT-12345',
});
```

You can then look up the Medplum user by their external ID when processing data from the external system.

See [External IDs](https://www.medplum.com/docs/user-management/external-ids) for full details.

## Custom invitation emails

By default, Medplum sends a standard invitation email. You can customize this email — including the subject line, body, and branding — using a Medplum Bot triggered on the invite event.

See [Custom Emails](https://www.medplum.com/docs/user-management/custom-emails) for configuration steps.

## Managing memberships

### Deactivating a user

To remove a user's access without deleting their membership record, set `active: false` on their `ProjectMembership`. This is the preferred approach because it:

- Takes effect immediately
- Preserves the audit trail
- Makes it easy to restore access later

Navigate to [app.medplum.com/admin/project](https://app.medplum.com/admin/project) > **Users** tab, select the user, and click **Remove User** — or update the resource via the API.

### Searching for project members

```typescript
// All project members
const allMembers = await medplum.searchResources('ProjectMembership');

// Only human members (exclude Bots and ClientApplications)
const humanMembers = await medplum.searchResources('ProjectMembership', {
  'profile-type': 'Patient,Practitioner,RelatedPerson',
});

// Only patients
const patients = await medplum.searchResources('ProjectMembership', {
  'profile-type': 'Patient',
});
```

## Promoting a user to admin

### Via the Medplum App

1. Navigate to [app.medplum.com/admin/project](https://app.medplum.com/admin/project) > **Users** tab.
2. Select the user you want to promote.
3. Check the **Admin** checkbox and save.

### Via API

Update the user's `ProjectMembership` resource and set `admin: true`:

```typescript
const membership = await medplum.readResource('ProjectMembership', membershipId);

await medplum.updateResource({
  ...membership,
  admin: true,
});
```

<Note>
Always spread the existing `ProjectMembership` when updating it to avoid accidentally clearing other fields.
</Note>
