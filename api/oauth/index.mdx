Medplum implements OAuth 2.0 and OpenID Connect (OIDC) for authentication and authorization. It also supports [SMART App Launch](http://hl7.org/fhir/smart-app-launch/) for FHIR-native application integration.

**Base URL:** `https://api.medplum.com`

---

## GET /oauth2/authorize

Initiates the OAuth 2.0 authorization code flow. The client redirects the user's browser to this endpoint. After authentication, Medplum redirects back to your `redirect_uri` with an authorization code.

**Request parameters:**

| Parameter | Required | Description |
|-----------|----------|-------------|
| `response_type` | Yes | Must be `code` |
| `client_id` | Yes | Your registered client application ID |
| `redirect_uri` | Yes | Must exactly match a pre-registered URI on the client |
| `scope` | Yes | Space-separated list of requested scopes |
| `state` | Recommended | Opaque value to protect against CSRF attacks |
| `code_challenge` | Conditional | PKCE challenge. Required when `code_challenge_method` is set. |
| `code_challenge_method` | No | PKCE method. Must be `S256`. |
| `nonce` | No | Random value embedded in the returned ID token |
| `prompt` | No | `none` (no UI, error if not authenticated) or `login` (force re-authentication) |

**Authorization code grant:**

```
GET https://api.medplum.com/oauth2/authorize?
    response_type=code&
    client_id=492e4ec3-fb66-4b45-b529-599c708ec530&
    redirect_uri=https://app.example.com/callback&
    state=random-state-value&
    scope=openid+profile+fhirUser
```

**Response (302 redirect to your app):**

```
Location: https://app.example.com/callback?code=AUTHORIZATION_CODE&state=random-state-value
```

**Authorization code grant with PKCE:**

```
GET https://api.medplum.com/oauth2/authorize?
    response_type=code&
    client_id=492e4ec3-fb66-4b45-b529-599c708ec530&
    redirect_uri=https://app.example.com/callback&
    state=random-state-value&
    scope=openid&
    code_challenge_method=S256&
    code_challenge=CODE_CHALLENGE
```

**Error responses** are returned as redirect parameters:

| Error | Cause |
|-------|-------|
| `invalid_request` | Missing or malformed parameters |
| `unauthorized_client` | Client not allowed for the requested `response_type` |
| `invalid_scope` | Unknown or invalid scope requested |
| `server_error` | Unexpected server error |

## POST /oauth2/token

Exchange an authorization code, refresh token, or client credentials for access tokens. All requests use `Content-Type: application/x-www-form-urlencoded`.

If your client has a secret, pass it via HTTP Basic Auth:

```http
Authorization: Basic Base64(<client_id>:<client_secret>)
```

### Authorization code exchange

```bash
curl -X POST https://api.medplum.com/oauth2/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -H "Authorization: Basic Base64(client_id:client_secret)" \
  -d "grant_type=authorization_code&client_id=492e4ec3-fb66-4b45-b529-599c708ec530&code=AUTHORIZATION_CODE&redirect_uri=https://app.example.com/callback"
```

**Response (200 OK):**

```json
{
  "access_token": "eyJz9sdfsdfsdfsd...",
  "id_token": "dmcxd329ujdmkemkd349r...",
  "token_type": "Bearer",
  "expires_in": 3600
}
```

### Authorization code with PKCE

```bash
curl -X POST https://api.medplum.com/oauth2/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=authorization_code&client_id=492e4ec3-fb66-4b45-b529-599c708ec530&code=AUTHORIZATION_CODE&code_verifier=CODE_VERIFIER&redirect_uri=https://app.example.com/callback"
```

**Response (200 OK):**

```json
{
  "access_token": "eyJz9sdfsdfsdfsd...",
  "refresh_token": "dn43ud8uj32nk2je...",
  "id_token": "dmcxd329ujdmkemkd349r...",
  "token_type": "Bearer",
  "expires_in": 3600
}
```

### Client credentials

For server-to-server (machine-to-machine) access:

```bash
curl -X POST https://api.medplum.com/oauth2/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -H "Authorization: Basic Base64(client_id:client_secret)" \
  -d "grant_type=client_credentials&scope=openid"
```

**Response (200 OK):**

```json
{
  "access_token": "eyJz9sdfsdfsdfsd...",
  "token_type": "Bearer",
  "expires_in": 3600
}
```

### Refresh token

```bash
curl -X POST https://api.medplum.com/oauth2/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -H "Authorization: Basic Base64(client_id:client_secret)" \
  -d "grant_type=refresh_token&client_id=492e4ec3-fb66-4b45-b529-599c708ec530&refresh_token=REFRESH_TOKEN"
```

**Response (200 OK):**

```json
{
  "access_token": "eyJz9sdfsdfsdfsd...",
  "id_token": "dmcxd329ujdmkemkd349r...",
  "token_type": "Bearer",
  "expires_in": 3600
}
```

<Note>
`refresh_token` is only issued when `grant_type` is `authorization_code` and the requested scopes include `offline` or `offline_access`.
</Note>

**Token endpoint error responses:**

```json
{
  "error": "invalid_grant"
}
```

| Error | Cause |
|-------|-------|
| `invalid_request` | Missing required parameter or malformed request |
| `invalid_client` | Client authentication failed |
| `invalid_grant` | Authorization code already used, expired, or refresh token revoked |
| `unauthorized_client` | Client not allowed for this grant type |
| `unsupported_grant_type` | `grant_type` is not one of the supported values |

## GET /oauth2/userinfo

Returns claims about the authenticated user. Requires a valid access token with the `openid` scope.

```bash
curl https://api.medplum.com/oauth2/userinfo \
  -H "Authorization: Bearer $ACCESS_TOKEN"
```

**Response (200 OK):**

```json
{
  "sub": "user-id",
  "name": "Jane Doe",
  "given_name": "Jane",
  "family_name": "Doe",
  "preferred_username": "jane.doe",
  "email": "jane.doe@example.com"
}
```

The `email` and `phone` values are sourced from the user's FHIR profile resource.

**Error responses:**

| Status | Error | Cause |
|--------|-------|-------|
| `400` | `invalid_request` | Missing or malformed Authorization header |
| `401` | `invalid_token` | Access token is expired, revoked, or invalid |

## GET /.well-known/openid-configuration

OpenID Connect discovery document. Clients can use this to auto-configure endpoints and supported features.

```bash
curl https://api.medplum.com/.well-known/openid-configuration
```

**Response:**

```json
{
  "issuer": "https://api.medplum.com",
  "authorization_endpoint": "https://api.medplum.com/oauth2/authorize",
  "token_endpoint": "https://api.medplum.com/oauth2/token",
  "userinfo_endpoint": "https://api.medplum.com/oauth2/userinfo",
  "jwks_uri": "https://api.medplum.com/.well-known/jwks.json",
  "response_types_supported": ["code", "token"],
  "subject_types_supported": ["public"],
  "id_token_signing_alg_values_supported": ["RS256"],
  "scopes_supported": ["openid", "profile", "email", "offline_access", "fhirUser"]
}
```

## PKCE

Proof Key for Code Exchange (PKCE) is required for public clients (e.g., single-page apps, mobile apps). It prevents authorization code interception attacks.

<Steps>
  <Step title="Generate a code verifier">
    Create a cryptographically random string of 43-128 characters using URL-safe Base64 characters (`[A-Z][a-z][0-9]-._~`).
  </Step>
  <Step title="Create the code challenge">
    Compute `BASE64URL(SHA256(code_verifier))` for the `S256` method.
  </Step>
  <Step title="Include in the authorization request">
    Pass `code_challenge` and `code_challenge_method=S256` in the `/oauth2/authorize` request.
  </Step>
  <Step title="Include in the token request">
    Pass the original `code_verifier` in the `/oauth2/token` request. Medplum verifies it matches the challenge.
  </Step>
</Steps>

## Scopes

| Scope | Description |
|-------|-------------|
| `openid` | Required for OIDC. Enables `id_token` and `userinfo` access. |
| `profile` | Basic profile claims (name, etc.) |
| `email` | Email address claim |
| `fhirUser` | FHIR resource reference for the authenticated user |
| `launch` | EHR launch context |
| `launch/patient` | Patient context in standalone launch |
| `patient/*.read` | Read all resource types in the patient's compartment |
| `patient/*.write` | Write all resource types in the patient's compartment |
| `user/*.read` | Read all resource types the user has access to |
| `user/*.write` | Write all resource types the user has access to |
| `offline_access` | Issue a refresh token for long-lived access |

## Token format

Access tokens and ID tokens are signed JWTs (RS256). The `id_token` contains standard OIDC claims:

```json
{
  "iss": "https://api.medplum.com",
  "sub": "user-id",
  "aud": "client-id",
  "exp": 1705315200,
  "iat": 1705311600,
  "nonce": "random-nonce",
  "fhirUser": "https://api.medplum.com/fhir/R4/Practitioner/practitioner-id"
}
```

Validate tokens using the public keys at `https://api.medplum.com/.well-known/jwks.json`.
